流程不是缺位，就是过载

两种极端，都会降低最终质量。

流程要么没有托底，要么成为新的负担。

TOO LIGHT不用 Skill：太轻
  • 直接写代码
  • 缺少方案与验收
  • 没有复盘和记忆
TOO HEAVY用了 Skill：太重
  • 每步确认、每事建档
  • 过度派 Agent
  • 上下文越压缩，执行越慢
流程本来为了降低风险最后却消耗了更多注意力

03/03

一条合格的 Workflow，必须又轻又稳

还要在长期使用中持续变聪明。

轻，不是少做；稳，也不是多做。

LIGHT

足够轻

  • 小任务不被流程拖慢
  • 只有真实业务取舍才打断人
  • 同一个问题只运行一套方法
STABLE

足够稳

  • 复杂项目能够跨会话恢复
  • 正式计划只批准一次
  • 没有 fresh 证据，不宣布完成
EVOLVE

会进化

  • 项目经验进入项目记忆
  • 稳定偏好进入用户记忆
  • 有复用价值的方法才升级
轻任务无感，复杂项目有序，长期使用持续变聪明。

01/07

看清需求

先查事实，再决定要不要问。

重点解决什么把一句模糊表达，收敛成可验收的业务结果。
A · 人的最佳实践

讲清工作方式与期待

说明长期场景、注意力痛点、最终要改善什么；不必先替 AI 发明阶段、字段和技术方案。

B · AI 的最佳实践

主动补齐项目上下文

调查现有 Workflow、历史规则、项目文档和宿主能力；把需求、事实、推断、未知与授权分开。

C · 常见错误与痛点

让用户反复补背景

把可查的目录、术语和工程现状问回用户；问了很多局部问题，仍没有形成完整业务目标。

D · 阶段成果

完整需求 baseline

目标、验收、非范围、未知分流与授权被写成一张可继续工作的确认卡。

人的原话“多个网页项目长期协作；人专注需求、方案、验收。”
→
AI 的确认卡“交付通用 Workflow + 项目记忆；网页项目作强验收样本。我将按此继续。”

02/07

方案选型

选定最终做成什么样。

重点解决什么从用户结果倒推，选出最小充分、最值得的路线。
A · 人的最佳实践

选择协同体验，不选字段

判断要“固定严谨 SOP”“自适应薄编排”还是“轻量提示词”；不需要决定状态名和模板结构。

B · AI 的最佳实践

先画出 90 分协同体验

比较人类打断次数、复杂项目可靠性、工程质量、维护成本与宿主兼容；给出明确推荐和删除项。

C · 常见错误与痛点

把严谨等于流程更长

每个任务都建文档、派 Agent、逐阶段确认；系统看似完备，用户却为协作制度持续买单。

D · 阶段成果

自适应薄编排方案

不仅选中 B，还讲清为什么删 A/C、保留哪些硬门，以及失败时怎样退回。

方案 A

固定全流程 SOP

每次走完整流程；可靠，但小任务也持续付流程税。

方案 B已选 · 推荐

自适应薄编排

只保留业务硬门；其余能力按风险和需要加载。

方案 C

轻量提示词手册

启动最快；跨会话、授权和完成判断依赖临场发挥。

03/07

计划编排

把路线编译成工作全景图。

重点解决什么拆成低冲突、可验收的业务 Plan 与可执行 Task。
A · 人的最佳实践

批准业务全景，不批施工细节

确认最终得到什么、哪些结果先后依赖、怎样算完成；整份正式计划只批准一次。

B · AI 的最佳实践

按用户结果拆 Plan

把“行为基线、协同协议、项目记忆、验证传播”编成可验收结果；复杂 Task 才展开执行合同。

C · 常见错误与痛点

计划像工程流水账

只有文件和命令，没有业务产出；并行与汇合看不懂，还要求人逐 Task 审批。

D · 阶段成果

正式提案 + 可执行合同

Plan 讲业务分叉汇合；Task 补齐目标、输入、输出、验收、写域与依赖。

P01 · 先定义锁定跨项目行为与验收样本
P02 + P03 · 可并行协同协议 · 项目 AI 记忆体系
P04 · 最后汇合验证、安装与讲座传播

◆

开工门

批准之后，才建立实现现场。

重点解决什么当前授权与工作现场，是否允许第一次产品写入？
A · 人的最佳实践
零参与

不用决定分支名

正式计划已经批准；本地安全现场由 AI 自行检查，不再问一句“是否继续”。

B · AI 的最佳实践

复用并校验安全现场

确认当前 Request、最新基线、项目入口和写域；现有 clean worktree 足够就复用，不制造第二现场。

C · 常见错误与痛点

隔离变成第二次立项

批准前就建现场，或宿主已有 worktree 仍再建一份；保护机制反而制造等待。

D · 阶段成果

Request 实现现场

批准版本、目标基线、写域和授权逐项对齐；不把“隔离”误做成第二次立项。

业务计划可开工+现场保护用户改动+基线仍支持计划=进入执行建设

04/07

执行建设

持续做，不做流程表演。

重点解决什么在授权写域内，把当前工作项做到并证明局部正确。
A · 人的最佳实践
零参与

不回复“继续”，也不盯 Agent

人在方案和计划上已经发力；正常实现期间，把注意力留给下一次真正判断。

B · AI 的最佳实践

按合同持续交付

同一写域只有一个 owner；先补行为场景，再改协议、四真源与介绍。Agent 只在净收益明确时出现。

C · 常见错误与痛点

角色很多，产物很少

多个 Agent 同改一页、多个 Reviewer 审同一 diff；同步成本超过实现本身。

D · 阶段成果

可运行的 Workflow 包

每个完成项同时带实际 diff、局部证据、残余风险与下一安全动作。

AI 持续推进查背景、写协议、补模板、跑场景、维护 progress
+
人在场次数零；没有新业务取舍就不发送“继续吗”

05/07

质量验收

从结果和 diff 重新判断。

重点解决什么证明结果简单、正确、完整，而不是相信“代码写完”。
A · 人的最佳实践

判断验收条件是否成立

只看轻任务是否轻、复杂项目是否可恢复、关键决定是否清楚；不用检查模板字段或运行命令。

B · AI 的最佳实践

用真实协作场景验行为

验证轻任务、正式项目、需求变更、宿主兼容与 clean-room 安装；复用未失效证据，只补缺口。

C · 常见错误与痛点

把用户当测试工程师

发一句“你看看有没有问题”；或 Task、Plan、最终结果机械重跑同一份证据。

D · 阶段成果

业务验收 + 证据矩阵

每一项用户承诺都映射到场景、基线、结果与原始证据，不用“测试绿了”替代业务完成。

人的验收条件“我只在需求、方案、计划 / 验收与外部授权处投入注意力。”
✓
AI 的证据轻任务零仪式；复杂项目可恢复；变更只重开受影响路径

06/07

集成交付

后合并的人，负责守住前人的成果。

重点解决什么把自己的改动，重新建立在最新主线之上。
A · 人的最佳实践
零参与

兼容意图时，不当冲突裁判

只有双方目标真的互斥，或需要 push、merge、上线授权时，人才出现。

B · AI 的最佳实践

先读前人意图，再 rebase

fetch 最新目标，比较主线与自己的起始基线；阅读前人 Plan、diff 与验证，把双方兼容意图重新整合。

C · 常见错误与痛点

整文件选边，等于覆盖前人

只保留“我的版本”或“最新版本”都会丢功能；文本没冲突，也可能存在语义冲突。

D · 阶段成果

兼容双方意图的最新候选

冲突按业务意图消解；自己的验证与前人受影响路径都 fresh 通过。

01 · FETCH拉最新主线
02 · READ读前人 Plan / diff
03 · REBASE重放自己的改动
04 · RESOLVE按双方意图解冲突
05 · VERIFY双方路径都重验

07/07

复盘升级

把本轮经验，送回正确的记忆层。

重点解决什么这次学到什么，应该留在项目、用户，还是 Workflow？
A · 人的最佳实践

确认哪些偏好真的长期稳定

例如“主页面要短、深度按需展开”；一次字号调整和临时路径不值得记住。

B · AI 的最佳实践

先分类，再决定是否升级

项目事实、用户稳定偏好、跨项目方法各归其位；升级 Workflow 前先证明复用收益。

C · 常见错误与痛点

一次反馈长成一套制度

每次任务都写复盘、加字段、建文档；久而久之，维护流程再次超过交付价值。

D · 阶段成果

可循环进化的方法与记忆

候选经验按项目事实、稳定偏好、通用方法与一次性噪音分流；只有达门槛的才晋升。

项目记忆讲座结构、当前基线、验证与待整合状态
用户记忆主页面短、深度按需；英文克制；单 HTML 可分享
Workflow后合并者先 rebase，保全双方意图并双重验证
不沉淀一次性字号、临时路径与某轮 Agent 分工
