The operating unit

每次只让 AI 看见，做好当前一步所必需的内容。

不是把整个项目塞给每个 Agent，而是动态生成一个最小充分的 Workflow Cell。

一个 Cell，一项清楚的责任

执行合同写清任务，进度控制台保存状态；当前角色只收到即时生成的上下文包。上下文不够时，只补一个明确缺口。

工作对象一次交付、结果计划、执行任务或新反馈
当前阶段现在只回答哪一类问题
最小上下文相关事实、决策和证据引用
责任角色谁对这个结果负责
能力与权限怎样做，以及不能越过什么边界
证据与路由产出什么，凭什么进入下一步

One fact, one owner

四类真源，外加按需执行合同。

不是把所有背景塞进一个大文档，也不是让每个 Agent 建一套文件。每类信息只放在一个地方，执行时只取相关切片。

request.md · 看清问题

真实用户、背景、原因、需求、约束和验收基线。

findings.md · 保存依据

事实、来源、未知、专家结论和方案比较；有跨阶段复用价值时才创建。

plan.md · 确认方案

完整业务方案、多个结果计划、任务依赖、冲突预防和整体验收。

progress.md · 控制现场

所有对象当前阶段、下一动作、阻塞、反馈和证据索引只看这一处。

task.md · 按需派工

复杂、委派或跨会话任务才生成；一项稳定 Task 一份，不按 Agent 数量复制。

不再默认创建

证据流水文档、持久化上下文包、每 Plan 进度、每角色文档和 Markdown 日记。

One request, many active cells

一张进度全景，多个 Plan 各走各的阶段。

同一独立需求共享一个工作区，所有运行状态集中在一份 progress.md。总协调者只把当前步骤需要的内容交给对应 AI，计划时先消除写冲突，最终再统一汇合。

一个 Request

共同目标、需求账本、授权、工作区与最终验收基线。

progress.md · 唯一状态台分别记录 Request、Plan、Task 和反馈

多个并行 Cell

每条工作线只加载自己的上下文；有写冲突的工作提前串行。

P01 · 质量验收AI Reviewer · 只读实际 diff 与验收
P02 · 执行建设Task Owner · 只改授权文件域
P03 · 等待外部输入Blocker 不拖回未受影响 Plan
F04 · 看清反馈先分类，再决定回哪一阶段

一个汇合口

所有纳入 Plan 验收通过后，再同步最新目标、解决语义冲突并重跑整体证据。

Eight clear stages

每个阶段，只做每个阶段的事情。

选择一个阶段，查看它实际读取什么、由谁处理、怎样产生下一步。

STAGE 01

立项隔离

先保护现场，再从最新、正确的项目基线开始。

只读上下文请求、项目规则、Git 状态

不加载方案历史和交付细节。

责任角色总协调者

判断是当前需求的补充，还是独立新需求。

主要动作同步最新代码 + 独立工作区

一个独立需求一个工作区；AI 协作者默认共享。

输出安全基线

需求编号、起点版本、分支、工作区和项目入口。

STAGE 02

看清需求

一串要求分别是什么，它们怎样关联，什么才算做成？

只读上下文用户原话、项目事实

不把可查事实重新问给用户。

责任角色需求负责人 / 关键追问主持

把问题压缩成真正会改变方向的决策。

按需能力关键追问与变化需求分流

问题集中问、总数不超过 15；新反馈先分类。

输出需求账本与验收

稳定编号、关系、边界、未知和可观察成功标准。

STAGE 03

选定设计

怎样得到 90 分终局，同时删除不必要的复杂度？

只读上下文目标、证据、约束

不继承执行流水和无关历史。

责任角色动态专家森林

独立发散、交叉质询，再由方案 Owner 收敛。

按需能力体验、视觉与动效

UI 先完成完整状态和方向选择，非 UI 不加载。

输出推荐方案

终局、边界、不变量、删除项、回滚点和设计契约。

STAGE 04

计划编排

怎样把结果拆成可串并行、可交接、可独立验收的工作？

只读上下文选定方案、真实目录和命令

不把专家完整发言带入实施。

责任角色结果计划负责人 / 总协调者

建立结果计划和任务的前后依赖，指定唯一写入负责人。

并行原则先消除冲突，再并行

文件、代码位置、数据结构、生成物和共享资源重叠就串行。

按需补地基只补当前交付缺的最小入口

AI 项目入口、运行/部署、全景导航、DESIGN.md 四选所需，不铺治理体系。

STAGE 05

执行建设

当前最小可验结果是什么，怎样用最少改动把它做成？

只读上下文一份 Task 执行合同

再加当前状态切片和精确引用，不读取完整 Portfolio。

责任角色Task Owner

只改授权写域，返回实际产物和证据。

按需能力根因 Debug

只有真实失败才加载；先复现与证伪，不堆补丁。

输出Task 结果

diff、命令、失败、fresh evidence 和下游新事实。

STAGE 06

质量验收

能否更简单，以及凭什么证明做成了？

只读上下文验收、实际变更、相关证据

复核者不继承实现者的完成判断。

责任角色AI 独立复核者

不默认等待人类；无独立 AI 时同一模型切换职责。

按需能力复用已有实现 + 代码瘦身

复用、删除、合并、效率检查后再重新跑验证。

输出当前结果计划的通过证据

执行任务、结果计划和残余风险逐层说清。

STAGE 07

集成交付

怎样把已验证成果安全放到最新目标分支和真实环境？

只读上下文目标分支、变更意图、授权

不重新加载 Grill 和专家会议。

责任角色Integrator

总协调者唯一负责 Git 生命周期和交付状态。

按需能力Fetch / Rebase / 语义解冲突

冲突后旧证据失效，必须 fresh 重验。

输出真实交付状态

本地、提交、推送、合并或部署到哪一级，如实报告。

STAGE 08

复盘升级

这次哪里偏了，怎样让项目和 workflow 下一次更省力？

只读上下文计划与实际差异

不复制完整日志和成功叙事。

责任角色Review Owner

找返工根因、早期信号和应删除的机制。

按需能力受控进化

两次复现，或一次严重且可复现，才晋升 workflow。

输出No-op 或最小改进

项目事实、用户偏好和 workflow 规则进入各自唯一真源。

Two common realities

需求不会整齐地来，反馈也不会等流程结束。

一次提出一串需求

  1. 逐项接住：生成稳定 Requirement ID。
  2. 先查事实：把能发现的内容从问题中删除。
  3. 集中 Grill：独立问题并行问，最多 15 个。
  4. 建立关系：识别重复、依赖、冲突和共享基础。
  5. 编排结果：组成 Plan DAG 与 Task DAG，再决定串并行。

验收中又冒出新要求

  1. 冻结基线：保留原来怎样算完成。
  2. 批量收集：先进入 progress.md，分类前不改代码。
  3. 先改定义：必要时升级需求基线或方案版本。
  4. 保护交付：只失效受影响 Task，其他工作继续。
  5. 重验影响面：重新计算可执行任务和集成路径。

One prompt, install or update

一句话交给 Agent，首次安装，以后自动更新。

不需要 Python，也不需要先知道 skills 目录；Agent 会根据真实环境备份、同步并校验。

Agent 安装与更新提示词

同一句提示词同时覆盖首次安装和后续更新；旧版本会先备份，失败时恢复。

请从 https://github.com/qzl0215/workflow 安装或更新 workflow：自动找到你当前实际使用的 skills 目录，有旧版先整体备份，再以最新 main 中的 SKILL.md、references/ 和 templates/ 完整镜像替换，校验入口与本地引用后报告版本、来源 commit 和安装位置；任何一步失败都恢复备份，不要猜固定目录。
