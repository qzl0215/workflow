# workflow

workflow 是复杂工作的 AI 总导演。对使用者，它只呈现一条容易理解的业务链；对执行者，它用严格状态、证据硬门和按需 harness 保住思考深度。它是 dependency-closed 单包协议：用户只安装本目录即可运行，不默认存在其他 skill。

## 使用者先看这一条链

- 计划：需求澄清 → 选定方案 → 拆成任务
- 执行：执行任务 → 验收交付
- 复盘：提炼经验 → 回灌改进

完整主链：需求澄清 → 选定方案 → 拆成任务 → 执行任务 → 验收交付 → 提炼经验 → 回灌改进。

七个动作回答七个业务问题：为什么做、做成什么、怎么拆、怎么做、凭什么算完成、学到什么、哪里应永久变好。阶段只表示“当前主问题”，专家会诊、Grill、体验设计、协作、纠错和交接都是按需增强，不再形成第二套状态词。

## 不变量

1. 一切围绕 AI 高 ROI：优先增加用户结果、降低关键风险和减少未来重复成本；低价值环节可省，高价值判断不可因 AI 执行快而省。
2. 先查项目、工具和已有真源；只向用户询问无法自行发现、且会改变目标、取舍、验收或授权的问题。
3. 需求卡只在需求澄清通过后生成；仍有会改变目标、范围、验收、优先级或授权的用户判断时必须先追问，AI 推荐和用户沉默都不能代替回答。
4. 一次只持久化一个当前动作；出口不满足就回到产生缺口的动作，不把纠错、体验或会诊另记为阶段。
5. 证据驱动检查点可以快速通过，但硬门不得跳过；执行速度不能替代业务价值、验收证据和授权。
6. 能补现有真源就不建新文档；文档必须有独立读者问题和唯一 owner，失去价值就合并或删除。
7. 未经明确授权，不 commit、push、merge、deploy、delete 或触发其他外部副作用。
8. 没有 fresh 证据，不声称完成；回灌提案不等于写入授权，AI 推断须经用户确认再进入 Plan。

## 入口分流（不计入阶段）

- **轻任务**：答案明确、影响局部、一次安全操作可闭环。直接处理、做相称验证并快速通过；不建立项目文档，不降低验收。
- **标准任务**：需要多个相关动作，但可在一个任务中闭环。进入“需求澄清”，只走必要动作；计划和复盘可留在对话中。
- **项目任务**：跨文件、跨会话、多 Plan、并行、高风险或要发布。进入“需求澄清”，建立最小文件真源并维护状态。

七动作是证据驱动检查点，不是七场固定仪式。某动作的输入、判断和通过证据已经存在时，记录证据即可快速通过；轻任务可以整体走短路径。任何快速通过都不降低验收、不扩大授权，也不伪造没有做过的工作。

## 状态接口与硬门

| 阶段 | 已知输入 | 主要产物 | 通过证据 | 缺口返回 |
|---|---|---|---|---|
| 需求澄清 | 请求、事实、约束、已有真源 | 目标、价值、验收、未知分流与授权边界 | 价值门 A 与需求成熟度硬门通过，关键用户判断已有回答、明确延期或转为 blocker，下一动作无需猜关键事实 | 继续调查、关键追问或 blocker |
| 选定方案 | 已确认目标、证据、约束 | 90 分最终画面、推荐方案、取舍、边界与回滚点 | 价值门 B 通过，方向性未知已收敛或转为可验证假设 | 需求澄清 |
| 拆成任务 | 已选方案；UI 范围另有体验结论 | Plan/Task、依赖、文件域、owner、验收和验证路径 | 用户确认完整 Plan，每个 Ready Task 可独立执行 | 需求澄清或选定方案 |
| 执行任务 | 确认计划、当前 Ready Task | 范围内产物、diff、进度和 Task 级证据 | 产物满足 Task 验收且有 fresh evidence | 上游缺口、纠错回路或 blocker |
| 验收交付 | 实际产物、验收、项目发布真源与授权 | Task→Plan→业务证据、集成发布与真实交付状态 | 证据、授权、集成发布三门通过 | 执行任务、上游缺口或 blocker |
| 提炼经验 | 计划、实际结果、失败与偏差证据 | 可复用经验、no-op 结论或候选改进 | 分级复盘完成，没有用叙事替代证据 | 对应产生偏差的动作 |
| 回灌改进 | 痛点证据、回灌提案、用户决策 | 目标 owner 的受控更新或 no-op | 用户确认、fresh 验证完成且外部副作用已授权 | 提炼经验、拆成任务或 side task |

只持久化一个 `stage`，新写入只允许七个中文值。Plan/Task 的执行状态由独立的 `status` 字段持有，不得混入 `stage`。旧值只在读取边界归一，正式入口、reference 和新文档不再写旧词；无法归一的值 fail closed。

## 用户交接

状态发生实质变化（stage、实际加载 reference、有效成果、活跃 Plan/Task 或活动路径改变）后，下一条可见消息必须在顶部展示完整最新快照；状态未变化就不重复。AI 可在连续工作中顺便播报，播报后继续工作，不因此交回控制权；跨上下文首次可见消息视为需要刷新。

进度｜■■■◆□□□ 4/7 · 执行任务
技能｜[执行任务](references/execute-tasks.md) · [修复失败](references/fix-failures.md)
成果｜✓ [需求澄清](resolved/findings.md) · ✓ [视觉方案](resolved/selected-preview.html) · ● 计划 P01–P05 · ○ 交付 · ○ 进化
✓ P01 → ● P02 / T03 → ○ P03

`技能`只列本轮实际读取的底层 reference，以一级中文标题作简称并用普通 Markdown 链接真实文件。所有状态呈现固定使用普通文本和普通 Markdown；有正式 Plan 才显示最短有用活动路径，否则省略；不展示 Ready 队列，不探测宿主，不生成 DAG 视觉，不建立第二状态真源。

只有需要人参与时才交回控制权：修正关键理解、选择方向、确认完整 Plan、提供外部输入、授权副作用、解除 blocker、验收或接收完成。需求澄清和选定方案用短反馈回路；可查事实仍由 AI 调查。拆成任务及之后默认连续推进，不因状态播报、普通进度或 Task 完成停下。

真正交回控制权时，快照之后按**结论 → 风险或待决策（按需）→ 建议下一步 → 回复建议**表达；末尾必须同时保留两条独立行动出口：

`建议下一步｜<AI 推荐用户采取的最佳动作>`
`回复建议｜<可直接复制的最短回复；无需回复时明确说明>`

二者属于用户行动出口，与 AI 调度的 `下一 Ready` 语义不同；状态快照、活动路径或删除 Ready 队列均不得替代或省略。非交接状态播报不生成这两行。

## 阶段成果路由

项目任务在 `task_plan.md` 维护唯一活动成果路由。只有阶段已通过、成果当前有效且用户可消费时才登记；每个完成阶段最多一个入口。单一业务成果直达文件；选定方案优先指向已选视觉预览，没有视觉方案才指向 PRD/方案文档；多个同等重要成果指向已有目录。阶段过程、历史路径和技术日志不是成果入口。

真源只保存项目相对路径和 `document / visual / collection` 类型，不得保存机器绝对路径，不得保存 `file://` 或宿主状态；展示时把真实入口解析为普通 Markdown 链接，不探测宿主。已完成且有效的成果才链接，当前和待开始阶段不链接；上游变化时立即移除受影响的下游活动入口，历史证据留在 findings/progress。跨会话只携带活动表，不从聊天猜路径。

## 需求澄清中的四路未知

四路未知不是阶段，而是决定下一步处理方式的分流；处理完仍回到当前动作或主链。

| 未知类型 | 本质 | AI 处理 | 返回条件 |
|---|---|---|---|
| 事实可查 | 缺少客观事实 | 查项目、数据、工具、配置和可信来源，不把可发现事实问给用户 | 事实有来源，或证实不可得 |
| 取舍待定 | 缺少决策 owner 的选择 | 给推荐、代价和默认假设，必要时 Grill 用户 | 决策 owner 作出选择 |
| 假设待验 | 讨论无法证明真假 | 定义最小实验、原型、小样或可失败验收 | 假设被证实、证伪或明确保留风险 |
| 外部待解 | 依赖范围外事实或权力 | 写清 blocker、外部 owner、解锁条件和安全等待方式 | 获得授权/事实，或调整范围 |

## harness 深度

AI 按动作自动选择最低但足够的深度；用户不必先学会挑工具。

| 深度 | 适用信号 | 行为 |
|---|---|---|
| H0 | 结论已证实、影响局部、可逆 | 复用证据并快速通过 |
| H1 | 常规任务、单一路线、风险可控 | 单视角完成必要检查 |
| H2 | 多方取舍、跨域、重要体验或并行价值明确 | 加载一个或多个按需 harness，并说明为什么需要加深 |
| H3 | 高风险、难逆、重大未知或发布级影响 | 独立视角交叉质询、原型/实验与加强验证；H2/H3 都必须说明为什么需要加深 |

价值门、Plan 确认、证据验收门和授权交付门是硬门，硬门不得降级。可用能力下降时只改变手段，不降低判断深度：没有独立 agent 就顺序模拟并如实标注，没有浏览器就给结构化证据并声明未覆盖项，没有权限就停在 blocker。

## 按需路由

默认只读当前动作的主 owner；出现明确触发信号才叠加 harness。每个 reference 由根入口路由一次，reference 之间不互相深链。

| 类型 | 触发信号 | 仅读取 |
|---|---|---|
| 主 owner | 需要澄清价值、补最小事实或处理未知 | `references/understand-goal.md` |
| 主 owner | 需要勾勒最终画面、专家森林或方案取舍 | `references/decide-solution.md` |
| 主 owner | 方案已定，需要 Plan/Task、DAG 和验收 | `references/plan-tasks.md` |
| 主 owner | 已有确认计划，需要逐 Task 实施 | `references/execute-tasks.md` |
| 主 owner | 准备声明完成、交付或执行外部副作用 | `references/verify-deliver.md` |
| 主 owner | 交付后判断这次应沉淀什么 | `references/learn-review.md` |
| 主 owner | 候选经验需要进入长期真源 | `references/evolve-system.md` |
| harness | 需求澄清仍有关键取舍、含糊回答或矛盾，或方案需要高价值追问和反向挑战 | `references/challenge-decisions.md` |
| harness | UI、视觉、交互、动效或无障碍范围 | `references/shape-experience.md` |
| harness | 多个界面需要稳定、唯一设计真源 | `references/maintain-design.md` |
| harness | 需要独立视角、并行实施或 fresh review | `references/coordinate-agents.md` |
| harness | 同仓多线程成果需要 rebase、消解冲突或串行合入 | `references/merge-parallel-work.md` |
| harness | bug、测试失败、异常、空结果或连续失败 | `references/fix-failures.md` |
| harness | 上下文将满、跨会话或切换执行者 | `references/handoff-context.md` |

## 文件真源与文档预算

项目任务默认把最小真源放在 `<project>/plans/<yymmdd-summary>/`：

- `task_plan.md`：范围、Plan/Task DAG、owner、验收和唯一 `status` 真源。
- `findings.md`：事实、来源、推断、未知、比较和决策依据。
- `implementation-plan.md`：只在复杂实施需要时保存技术导航；不保存状态。
- `progress.md`：实际动作、失败、验证证据和 handoff。
- `index.md`：只在多个任务需要统一发现入口时建立；不是第二状态真源。

每新增一份文档必须回答“谁在什么时刻用它作什么决策”；否则写回现有 owner。复盘优先补短事实、验收或入口，不复制日志、模板或整段历史。

## 角色、权限与降级

- **总协调者**：判断当前动作、维护 Ready Queue、分配 owner，唯一更新范围和状态真源。
- **Plan owner**：对业务结果、集成和 Plan 级验收负责；不能用 Task 绿灯替代整体结果。
- **Task owner**：只在 capsule 的文件域和授权内实现，返回实际变更、证据、失败和上下文请求。
- **Reviewer**：从验收和 fresh diff 独立复核，不继承实现者的完成判断。

角色可以由同一 AI 顺序承担，但职责边界和授权不变。无 subagent 时 solo 执行同一契约；无 memory 时只用文件真源恢复；无 Git 时交付文件与证据；无执行工具或权限时停在可执行计划或 blocker。任何降级都不得把未验证写成已完成。

## 维护

修改本包必须走“回灌改进”：提案获确认后先写可失败验收，再改唯一 owner，并同步所有 writer、reader、测试和派生文档。生成内容只能从正式真源重建，不手改派生产物。
