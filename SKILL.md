# workflow

workflow 是复杂工作的薄编排层。它对用户只承诺四件事：看清真正目标、在必要取舍处作出清楚决定、持续做到并验明白、把有复用价值的经验留给下一次。

本包 dependency-closed：只安装本目录即可运行，不要求外部 skill、脚本、独立 Agent、特定平台或 Python。宿主能力缺失时，本包提供最小 fallback；能力缺失只改变手段，不降低业务验收。

## 责任边界

Workflow 只拥有五类规则：

1. 当前处于哪个业务阶段，以及出现偏差时回到哪里；
2. 哪个事实或业务取舍必须交给用户决定；
3. `findings.md / task_plan.md / implementation-plan.md / progress.md` 四份真源的边界；
4. 本地写入、外部副作用与难逆动作的授权门；
5. 什么证据足以称为完成。

调查、方案设计、UI/体验、计划方法、Agent 协作、调试、代码整理、验证和 Git 操作的具体做法属于宿主能力。每个关注点同时只能有一个 method owner，按以下顺序选择：

1. 用户明确指定、当前可用且与本协议兼容的能力；
2. 宿主已有、与本协议兼容的原生能力；
3. 本包中一个最小 fallback。

兼容意味着：不创建第二份需求、计划或进展真源；不重复询问同一决定；不扩大授权；结果能够回写四份真源或对话内简单路径。业务阶段协议始终负责边界和输出；命中前两项后，不执行、不叠加该关注点的条件 fallback，也不再读取对应 fallback 文件。宿主缺失时，每个触发关注点只读取一个对应 fallback。不同关注点可以组合，同一关注点不得叠加两套方法。

## 默认走最短但足够的路径

- 先查可获得的项目、代码、数据、配置和可信来源，再决定是否提问。
- **轻任务预算**：必读根协议后，若目标、方案、授权和验收都清楚，直接实施并做一次相称 fresh 验证；不因“经过阶段”机械读取 reference，不建文档、不派 Agent、不追加计划确认，也不机械新建 worktree。
- **标准任务预算**：默认由当前 AI 连续完成；阶段只作内部检查点，不逐阶段向用户报到。只有当前判断确实需要细则时才读取一个对应 reference，状态只在实质变化、等待或交接前更新。
- **复杂任务加深**：文档、Agent、独立复核和更强隔离都必须说明它会改变哪个决定、并行哪个独立写域、降低哪项主要风险或补足哪份必要证据；收益不高于派发、等待、整合和维护成本时不用。
- 已有充分且仍适用的结论直接复用；证据足够就收敛，不为流程完整制造步骤。
- 用户已授权实施后持续推进到质量验收和授权内交付。只有新的业务契约变化、无安全默认的关键取舍、未授权难逆动作、真实外部阻碍或用户要求暂停时才停。

## 用户交互预算

### 必要 Grill

只有同时满足以下条件才暂停询问：答案会改变用户结果、范围、验收、授权或难逆风险；事实无法自行查得；不存在安全可逆默认。触发时先完成事实调查，再由一个提问 owner 用业务语言集中询问：

- 一轮围绕一个决策主题，通常 1–3 个短问题；
- 每题说明推荐、2–3 个互斥选项和答案会改变什么；
- 不问实现偏好、可查事实、重复问题或不影响结果的问题。

完整业务边界见 `references/01-grill.md`。

### 三个规定触点

1. **需求确认卡自动续行**：标准或正式任务在 01 结束时展示“需求确认卡｜我将按此继续”，只含为谁解决什么、最终得到什么、本次做/不做、怎样算完成和安全默认；不问“是否确认”，随后自动进入 02。显式调用 Workflow 的轻任务不单独展示卡，把需求理解并入最终一句汇报。
2. **真实方案取舍才停**：存在需要人承担的体验、成本、风险或难逆取舍时，用选择题、短表、线框或 demo 中最合适的一种呈现并等待；单一优势路线清楚且可逆时，直接推荐并继续。用户选过的方向不得在后续重复询问。
3. **正式计划只批准一次**：03 一旦选择 `plan_mode=formal`，无论草案在对话、安全计划区还是最终文件中，都必须保持 `approval_state=draft / approved / rejected / modified`；只有 `approved` 才能开工。一次批准覆盖需求 baseline、已选方案、范围和编排；`plan_mode=inline` 的简单任务为 `approval_state=not-required`，不增加批准仪式。批准计划不等于授权 commit、push、merge、deploy、delete、公开发布或其他外部副作用。

用户只要求计划时，交付计划后结束，不进入开工门。正式计划批准后若出现新的业务契约变化，只呈现受影响变化并形成新版本；不得重问未变化且已决定的部分。

## 四份唯一真源

需要跨会话恢复、长期持久化或多人协作时才创建文件；简单局部任务优先留在对话。文件有真实内容时才创建：

| 真源 | 唯一拥有 | 不保存 |
|---|---|---|
| `findings.md` | 完整需求 baseline、背景事实与来源、推断、未知、方案判断依据、用户决定 | 运行状态、任务流水 |
| `task_plan.md` | 正式方案、PRD、Plan DAG、Task 拓扑、正式批准状态，以及 Plan 级范围、写 owner、依赖与验收 | 当前进度、验证日志、复杂 Task 的执行细节 |
| `implementation-plan.md` | 仅复杂、委派、跨会话或高冲突 Task 的同 ID 输入、写域、DONE、验证与停止合同；可展示从 `task_plan.md` 派生的只读 Plan DAG 与按 Plan 分组的 Task DAG | 独立维护的 Plan/Task 拓扑、第二份业务范围、每 Agent 文档 |
| `progress.md` | 当前状态、工作进展、验收记录，以及下一步或等待原因 | 完整历史、第二份计划或需求 |

`progress.md` 只使用 `待做 / 进行中 / 等待中 / 已完成` 描述工作项，并维持“当前 / 工作进展 / 验收记录”三块当前快照。已处理反馈、已解除等待和过期记录从当前快照移除；历史交给 Git 或原始产物。

正式计划包含多个业务结果时，`task_plan.md` 先用 Plan DAG、业务摘要和顶层 Task 表展示依赖与汇合，并唯一拥有 Plan/Task 依赖与波次；复杂执行再由 `implementation-plan.md` 展示从它派生的只读 Plan DAG，以及按 Plan 分组的 Task DAG、并行波次、跨 Plan 边与同 ID 合同。图中每个 ID 必须能直接定位同 ID 摘要或合同。拓扑变化必须先改 `task_plan.md` 再更新派生视图。简单 Task 留在 `task_plan.md` 内；复杂 Task 晋升后，前者只保留 ID、业务摘要、所属 Plan、依赖、波次和合同链接，不为 Task 或 Agent 机械建文件。

只有总协调者更新四份真源。执行者返回实际改动、证据、偏差和缺口；复核者从验收、实际 diff 与原始结果开始。任何宿主能力都必须使用这组文件名，不生成兼容副本。

## 七阶段业务路由

阶段是业务问题索引，不是运行清单或七次用户会议。根协议与现有证据足够判断时，可在一个连续执行回合快速通过多个阶段；不得仅为标记“走过”而逐个读取文件、汇报或写状态。只有某阶段成为当前实质问题且需要细则时，才读取表中对应协议。

| 阶段 | 只解决什么 | 详细协议（需要时） |
|---|---|---|
| 01 看清需求 | 真正要解决什么，怎样算完成？ | `references/01-understand.md` |
| 02 方案选型 | 做成什么样，哪条路线最值得？ | `references/02-design.md` |
| 03 计划编排 | 怎样拆成低冲突、可验收的工作？ | `references/03-plan.md` |
| ◆开工门 | 当前授权与现场是否允许首次实现写入？ | `references/start-gate.md` |
| 04 执行建设 | 当前工作项怎样做到并证明局部正确？ | `references/04-build.md` |
| 05 质量验收 | 当前结果是否简单、正确、完整？ | `references/05-verify.md` |
| 06 集成交付 | 怎样基于最新目标安全落地？ | `references/06-deliver.md` |
| 07 复盘升级（有信号才进入） | 什么应留下，哪里应永久变好？ | `references/07-review.md` |

路由不直接篡改不兼容的当前工作：目标、验收或需求关系错误回 01；方案或体验方向错误回 02；范围、依赖、写域或编排错误回 03；实现缺陷回 04；证明不足回 05；集成与交付问题留在 06；长期改进属于 07。未受影响工作保持当前结果和有效验收。

## ◆开工门

开工门不计入阶段编号；首次产品写入前必须成功通过。未通过或基线变化后可重新进入，但不重新讨论未受影响的需求或方案。

- `plan_mode=formal`：不看草案存在哪里，当前版本必须为 `approval_state=approved` 才能建立实现现场；`draft / rejected / modified` 一律返回 03。
- `plan_mode=inline`：用户原始明确执行请求就是本地、可逆、范围内写入授权，`approval_state=not-required`；不增加确认。
- 相称隔离：先复用 Codex thread、用户或兼容 harness 已提供且满足 Request 归属、隔离、最新基线与用户改动保护的工作现场，不重复创建 branch/worktree。正式计划、脏工作区、多人/并行、跨会话恢复、复杂依赖或较高风险工作在现有现场不足时，才基于最新可信目标创建一个 Request branch + worktree。局部、可逆、短生命周期小改可直接使用现有安全工作区。不得覆盖、stash、reset、清理或搬运用户现有修改；无 Git/remote 时采用当前环境可实现的最安全隔离并说明未覆盖项。
- 同一 Request 的执行者默认共享该 worktree，不自行创建分支、worktree、commit、rebase 或 push。重叠写域合并为一个 owner 或串行。
- 建立现场后重新核对 planning source fingerprint。基线漂移若改变用户结果、范围、验收、关键体验、难逆风险、成本/时间承诺或交付目标，当前正式计划批准失效并返回对应阶段形成受影响的新版本；若只影响 Task 依赖/波次、写域、owner、验证入口或可逆实现细节，总协调者先同步 `task_plan.md` / `implementation-plan.md` 和受影响验证，再以同一业务批准继续。
- 正式计划草案在批准前只放在对话或安全的任务存储；开工门通过后，若批准版本尚未位于复用或新建的 Request 工作现场才迁入，避免先污染用户目录或保留两份副本。

## 条件协议

业务边界协议用于决定是否停、回哪一阶段或能否完成；fallback 只在同类宿主能力缺失时读取。

| 所属 | 触发 | 协议角色 | 协议 |
|---|---|---|---|
| 01 | 无安全默认的关键业务未知 | 业务边界 | `references/01-grill.md` |
| 01 | 执行、验收或交付中收到新反馈 | 业务边界 | `references/01-delta-intake.md` |
| 02 | UI/旅程/视觉/交互方向需要方法支持 | fallback | `references/02-experience.md` |
| 02 | 多路线、跨域或难逆选择需要独立挑战 | fallback | `references/02-challenge.md` |
| 03 | 多 Plan、独立工作项或多个 Agent 需要协作方法 | fallback | `references/03-coordinate.md` |
| 03 | 当前目标缺少会直接阻碍执行的项目地基 | 业务边界 | `references/03-foundation.md` |
| 04 | bug、失败、异常、空结果或不稳定行为 | fallback | `references/04-debug.md` |
| 05 | diff 出现重复、复杂度、维护性或效率风险，或用户明确要求整理 | fallback | `references/05-simplify.md` |
| 06 | 需要 Git 集成或冲突修复方法 | fallback | `references/06-git-integrate.md` |
| 07 | 经验准备进入长期真源或修改本 workflow | 业务边界 | `references/07-evolve.md` |

## 证据、授权与完成

验收记录至少能回答：验了什么产物、基于哪些实质输入与环境、用什么命令或场景、对照哪个 baseline、何时观测、在什么条件下有效、原始结果在哪里。只有确定性检查、全部实质输入可识别、有效条件仍成立且原始结果可复核时才复用；任一实质输入变化，只重新验证受影响路径。外部可变事实必须在依赖它的决策点重新观测。

计划批准和本地实现授权不能替代外部授权。commit、push、merge、deploy、delete、公开发布、生产数据写入和其他难逆动作按目标逐项确认；未经授权保持可恢复现场并报告下一步。

只有纳入范围达到业务验收、必要集成基于最新目标通过、授权内交付动作真实完成且状态准确时才能称为完成。局部测试、模拟结果、计划文本、执行者自报或旧证据不能替代完成判断。

## Agent 与反馈边界

- 默认 solo；不能仅因 UI、多文件、多个阶段、概念角色或“多一个视角”派 Agent。只有 Agent 对应独立决策、独立写域，或能显著降低高风险误判，且收益高于派发、等待、整合与复核成本时才增加；收益消失立即收敛。同一 diff 与验收的 spec / 质量 / 效率 / 兼容审查默认合为一个正交终审，不按审查标题扇出重叠 Reviewer。
- 派发只提供版本化执行合同、`progress.md` 当前切片和精确事实/代码引用；不默认附完整聊天、全部历史或无关工作。
- 子 Agent 不改四份真源、不操作 Git 副作用；总协调者读取实际 diff 与结果后统一更新。
- 一串同一交付意图的要求属于一个 Request；先编号、查事实与关系，再形成 Plan DAG 和 Task DAG。可独立交付、验收和排期的弱相关目标另建 Request，并从最新目标经过自己的开工门。
- 新反馈先冻结原验收 baseline 并分类，分类前不改代码。原需求未满足回 04；方向变化回 02；范围或编排变化回 03；同目标小优化进入当前计划；独立结果建立新 Request。未受影响工作继续。

## 安装、降级与维护

Clean-room 安装或更新时，Agent 唯一定位真实 skills 目录，先解析一个不可变 commit SHA，再从同一 SHA 把 `SKILL.md`、`references/`、`templates/` 和按需维护契约 `tests/scenarios.md` 下载到临时目录并校验，最后原子替换；失败恢复备份或删除不完整的新目录，结束时报告版本、commit 和安装位置。全过程不得要求 Python。

没有独立 Agent、浏览器、Git、网络、持久 memory 或执行权限时，使用当前环境能做到的最安全手段并明确未覆盖项；不把计划、模拟或未验证结果写成完成。

修改本包时，先在 `tests/scenarios.md` 固化会失败的可观察行为，再改唯一 owner，并同步所有真实 reader、writer 与派生介绍。仓库保持零 Python、零生成注册表；直接检查 frontmatter、相对引用、可达文件、版本、安装提示词同步、敏感信息、冲突标记、行为场景与实际 diff。讲座和介绍只从最终协议派生，不保留已删除概念。
