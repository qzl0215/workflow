# 维护设计真源

## 何时进入

多个界面任务需要共享设计决策，或现有规则、tokens、组件与页面模式冲突，导致实现会重复猜测时进入。单页小改、一次性 demo 或非 UI 任务不自动建立设计系统。

## 已知输入

- 选定体验方向与可测试状态；
- 现有 DESIGN、tokens、CSS、组件、品牌规则和页面模式；
- 重复出现的决策、冲突证据、使用者和代码 owner；
- 预期跨任务寿命与维护成本。

## 深度判断

只有规则具备**稳定复用价值**、有明确读者/owner、且不沉淀会让后续反复猜测时才长期维护。优先补现有设计真源；能由代码 token 或组件表达的细节不再复制成文档。非 UI 任务不得创建 DESIGN 或空设计说明。

## 核心动作

1. 审计现有 tokens、CSS、组件、品牌与页面模式，定位真实代码 owner 和冲突。
2. 先合并、删旧或补现有真源；只有 UI/品牌范围明确、跨任务收益成立且没有 owner 时，才新建根级设计入口。
3. 文档只保存稳定决策：atmosphere/density、语义色彩角色、字体层级、间距/布局、组件/页面模式、响应式、motion、完整状态和 do/don't。
4. 代码 tokens 和组件实现继续承担数值与行为细节；文档链接真实 owner，不复制整套值。
5. 每条规则必须能映射到实现或验收，删除“高级、现代、干净”等无约束形容词。
6. 新规则先在当前体验中验证；没有第二使用场景的抽象保持局部，不提前平台化。

## 写入真源

优先修改现有 DESIGN、token、组件或品牌 owner；没有入口且晋升门成立时才新增一份短设计入口。体验选择仍留在任务契约，设计真源只保留跨任务稳定规则。

## 停止 / 通过

实现者能找到唯一设计 owner，并把体验结论映射到既有或新增 token、组件和页面模式；重复/冲突规则已清除；长期收益大于维护成本。单页局部结论能够留在当前任务时以 no-op 结束。

## 失败回路 / 能力缺口

规则仍只服务一次任务时退回体验塑形；代码与文档冲突时以实际实现和 fresh 检查定位差异，不能随意选一份。没有修改设计系统的授权或文件域时只记录候选与证据，不绕过 owner 写第二份说明。
