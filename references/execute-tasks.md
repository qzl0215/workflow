# 执行任务

## 何时进入

已有用户确认的 Plan、依赖已满足的 Ready Task、明确文件域、验收和验证路径时进入。默认一次执行一个可验收 Task 或最小垂直切片，以保持证据和文件域清晰。

## 已知输入

- 当前 Snapshot、Plan/Task ID、P9 六要素和相关 findings；
- 允许/禁止文件、共享资源、授权边界和回退点；
- 上次 handoff、已排除假设、实际命令与最易过期事实；
- 当前 source fingerprint、Task 绑定、执行现场、目标基线、freshness 来源、脏改动归属与 Task 级验收。

## 深度判断

执行深度由风险、可逆性、未知和影响面决定，不由代码行数决定。先判断 Task 是否仍在 Ready Queue、source fingerprint 是否变化、文件域是否被其他 owner 占用、是否出现范围变化。任何可变更 Task 在首次开工或跨上下文恢复时只做一次**执行现场有效性检查**；它是派生检查证据，不是新的 stage 或 status。只有两个以上独立 Task 文件域隔离、验证独立且并行 ROI 为正时才协调并行。

## 核心动作

**执行现场有效性检查**必须在 Task **标记进行中之前**完成：

1. 解析实际项目/工作现场、Task 绑定、项目规定的目标基线与 freshness 来源；使用项目真源提供的检查或版本机制，不在通用 workflow 硬编码 Git、remote 或平台命令。
2. 对照 source fingerprint，确认现场属于新 Task、同一活跃 Task、已被目标吸收的历史成果，还是身份不符/事实不可证；同时确认脏改动归属、文件隔离和可写性。
3. **新 Task**绑定是指新目标/新 Plan 的首个可变更单元，或不属于当前已批准计划链的工作；它只从项目认可的 fresh 基线和现场开始。项目要求隔离时必须新建隔离现场，不从旧 Task 或另一功能现场继续分叉。
4. **同一活跃 Task**或输入明确继承当前现场的**同一计划链的后续 Task**遇到目标前进不自动失效，也不能仅凭 `behind > 0` 强制 rebase；记录漂移与重叠风险，按项目契约决定同步点，并在交付前吸收最新目标。
5. 现场**已被目标吸收**、绑定其他 Task 或来源不明时，只允许明确声明的**只读诊断**与追溯；新工作必须回到 fresh 基线，不能把历史现场重新绑定为可写 Task。
6. freshness、归属或可恢复性无法证明时 fail closed；安全的只读检查可继续，但必须声明所依据的缓存/时间边界和未覆盖事实。

执行**单 Task 循环**：

1. 标记进行中，冻结输入、source fingerprint、文件域、共享资源、授权和验证方法。
2. 先获得最小失败证据或缺口证据，再做满足验收的最小实现；适用时保留红→绿→整理证据。
3. 只改 Task 文件域；不在执行任务中偷做上游决策，也不清理或覆盖其他 owner 的工作。
4. 运行最小相关验证，再扩大到 Plan 相邻回归；记录实际命令、exit code 和关键输出。
5. 失败进入根因回路；同一路径连续失败时改变假设、观察点或手段，不机械重试。
6. 用 fresh evidence 对照验收后才完成 Task；source fingerprint 变化则证据作废并重跑。
7. 重算 Ready Queue，选择下一 Task 或上收 blocker。

**持续推进授权**：用户明确表示“持续推进 / 直到完成 / 不需要我决策就继续”时，这是一项跨 Task 的执行授权，不是只执行当前切片。每个 Task 验证并写入真源后，自动进入下一个 Ready Task；必要时在同一条对话中简短报告进度，但不因普通进度或 Task 完成停下，也不得因单个 Task 完成而结束整体工作。只有所有纳入 Plan 验收完成、出现真实 blocker、需要新的业务决策，或用户明确暂停/缩小范围时才停止。

**范围变化分流**：原验收未满足时重开原 Task；验收修正且不改变方案的留在当前 Task；新的独立用户结果回到拆成任务追加 Plan；改变目标/方案/体验的回到对应上游动作；外部副作用停在授权门。

## 写入真源

产品实现写入 Task 文件域；执行现场检查、实际动作、失败、验证和 handoff 写入 `progress.md`；范围和状态只由 `task_plan.md` 更新。现场证据不复制成第二状态，不得把实现细节复制进多个计划文件，也不得在派生产物中修正规则源。

## 停止 / 通过

Task 输出、diff、fresh evidence、残余风险和下一步均可由另一位执行者复核；验收全部满足、无越界文件和未授权副作用。完成后进入验收交付；出现真实偏差则进入根因回路或上游动作。

## 失败回路 / 能力缺口

第一次失败读取错误并验证根因假设；第二次换本质不同方案；第三次扩大证据来源、反转假设或请求外部决策。无执行能力时交付可运行计划和 blocker，不声称实现；无 agent 并行能力时 solo 执行；共享资源被占用时安全等待或改走独立 Task，不轮询制造噪音。
