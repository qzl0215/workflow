# 04｜执行建设

一次推进一个可开始的工作项，或一组已经证明互不冲突的工作项；本阶段守住合同、写域、授权和局部 DONE，具体实现方法由当前宿主能力负责。

## 最小上下文

- 当前 Plan/Task ID、目标、输入、输出、验收、依赖和 source fingerprint；
- 允许修改、只读和禁止的文件/符号/共享资源；
- 相关项目事实、代码、测试入口、已排除假设和回退点；
- ◆开工门确认的 Request 工作现场、实际隔离方式与当前授权。

简单工作使用对话内合同；复杂、委派或跨会话工作从 `implementation-plan.md` 同 ID 合同、`progress.md` 当前切片和精确引用生成最小上下文。不默认读取完整 Request、全部方案讨论或无关 Task 历史。

## 方法 owner

先使用一个用户指定或兼容的宿主原生实现能力；Workflow 只检查它是否遵守范围、写域、授权和回写契约。遇到真实失败时，再按根协议为“调试”这一独立关注点选择一个 owner；用户指定或宿主原生调试能力存在时不读取 `references/04-debug.md`，缺失时才把该文件作为唯一 debug fallback。

## 阶段能力

1. 开始前核对当前 baseline/计划版本、工作项合同、依赖、写域占用、source fingerprint 和验证入口；失效就回 01–03 修正唯一真源，不猜测继续。
2. 先取得最小失败或缺口证据，再做满足验收的最小实现；适用时保留红→绿→整理证据。
3. 只改授权写域。发现上游缺口、业务契约变化、共享资源冲突或需要新外部副作用时停止当前写入并路由，不顺手扩大范围。
4. 每个工作项完成后运行最小相关检查和受影响相邻路径；记录被验产物、实质输入、环境、命令/场景、baseline、观测时间、关键结果与原始位置。
5. bug、失败、异常、空结果或不稳定行为才触发调试；没有真实失败时不得读取 debug fallback。修复必须针对根因并补相称回归。
6. 执行者结构化返回实际产物与 diff、检查结果、失败、已排除假设、新事实、偏差、上下文缺口和推荐路由；不能自行宣布 Plan 或 Request 完成。
7. 总协调者读取实际结果后，只在工作项状态、业务结果、验收、等待原因发生实质变化，或准备交接/切换上下文时更新 `progress.md`；不按工具调用或细碎步骤写流水，简单路径留在对话。已授权持续推进时，自动选择下一个依赖满足且无冲突的工作项，直到纳入范围完成、出现真实等待、需要新业务决定或用户暂停。

## 输出与路由

- 工作项产物、实际修改文件/符号和 diff；
- 定向检查、失败与已排除假设、残余风险；
- 新发现、范围偏差、上下文缺口和下一工作建议。

当前 Plan 的纳入工作全部具备可复核产物后进入 **05 质量验收**。实现缺陷留在 04；需求/验收错误回 01，方向错误回 02，范围/依赖/写域错误回 03。新反馈先进入 `references/01-delta-intake.md`，分类前不直接修改实现。
