# 02｜看清需求

把用户的一句或一串表达沉淀为可验证的结果、需求关系和最贵未知；不在此阶段编实现 Task。

## 最小上下文

- 用户原话、已表达偏好、Request 摘要和已有 Requirement Ledger；
- 项目规则、相关入口、已有行为、数据和可信来源；
- 当前事实、推断、约束、非范围、授权和最易过期信息；

不默认读取实现流水、完整仓库历史、交付协议或其他 Plan 的证据。

## 责任角色

**需求 owner**负责把可查事实与主观取舍分开；**总协调者**唯一更新 `request.md` 当前 baseline、`findings.md` 和 `progress.md` 的 Feedback Queue。需要用户回答时由总协调者合并问题，不让多个 Agent 分别追问。

## 阶段能力

1. 将每项原始要求赋予稳定 Requirement ID，改写为用户/场景/当前→目标/可观察验收/非范围。
2. 判断需求之间的重复、包含、依赖、冲突、共享基础、可并行、互为备选或无关，形成关系图而非一句话一个 Task。
3. 未知只按处理方式分流：事实可查由 AI 调查；取舍待定进入 Grill；假设待验定义最小实验；外部待解写 owner 和解锁证据。
4. 只读检查最短相关路径；能从项目、工具、数据、配置或可信来源获得的事实不问用户。
5. 多项关键取舍触发 `references/02-grill.md`；执行或验收中出现一批新反馈触发 `references/02-delta-intake.md`。
6. 继续调查或提问已不再改变目标、Plan 分组、主要顺序、验收、授权或难逆风险时立即停止。

## 输出

- Requirement Ledger 和需求关系图；
- 总体用户结果、成功信号、约束、非范围和授权边界；
- 带来源的事实、明确推断、四路未知及其闭环方式；
- `request.md` 中可确认的需求与验收 baseline，以及交给 03 阶段的精确 `findings.md` 引用。

## 退出与路由

关键事实无需猜测，需求关系足以决定是合并、依赖、并行还是拆分 Request 后路由到 **03 选定设计**。只有单一路径、低风险、可逆且方案已有证据时可快速通过设计检查进入 04；事实缺失继续调查，关键取舍加载本阶段 Grill，外部权限缺口保持 blocked。
