# 回灌改进

## 何时进入

复盘产生了有证据的痛点、稳定偏好或高价值改进可能，或正在修改 workflow 本包的入口、reference、template、script、test、生成文档时进入。它把 AI 猜出的补充需求变成用户可决策的回灌提案，再把获批方案交给正确 owner。

## 已知输入

- 真实失败、重复成本、偏差或用户偏好的事实证据，以及仍属模型推断的部分；
- 候选改进的影响面、最小改造、预期价值、唯一 owner、失败验收和维护成本；
- 当前项目、用户级稳定真源、目标 skill 的 writer、reader、测试与兼容边界；
- 用户对提案与完整 Plan 的决策，以及外部写入、发布或跨项目变更的授权状态。

## 深度判断

是否值得提案由**模型判断**，不设置出现次数或改进配额。综合证据可信度、实际伤害、未来影响、改动大小、可逆性和长期 ROI；一次发现可以产生高价值提案，重复出现的低价值噪音也可以不提。

提交前做**奥卡姆删除测试**：删掉一半是否仍解决问题、能否只改现有 owner、是否真是“小改动、大价值”、新增规则是否反而增加上下文负担。不能通过就 no-op，不为证明复盘有用而制造回灌。

提案门与写入门分开：模型可以主动提出需求，但不能把推断直接变成长期规则。用户级承载跨项目稳定偏好，项目级承载当前项目事实与路径，skill 级承载可复用方法；workflow 归入 skill 级。

## 核心动作

1. 生成最小提案，只保留**痛点问题、推断需求、最小改造、预期价值、接受 / 调整 / 暂不做**；事实与推断分开，价值用产出质量、交互次数、无意义 token 和上下文质量表达。
2. 用户接受后固化需求、方案、非范围和验收；**提案接受只授权进入计划**，不得自动写入。完整 Plan 仍需用户确认，外部副作用仍需明确授权。
3. 先分清缺陷还是新需求：**原验收未满足**就修正原 P/T；只有**新的用户结果**才追加新 Plan，不能用“回灌”掩盖漏做。
4. 按 owner-local handoff 路由：目标 owner 在当前上下文时做**零距离 handoff**并进入“拆成任务”；目标在其他项目或 skill 时生成 **side-task capsule**，来源任务保持“回灌改进”，输入充分的目标任务从“拆成任务”开始。
5. 用户级或无项目场景先找现有稳定 owner；**无 owner**或 owner 冲突时 fail closed，请用户决定，不随意创建长期文件。
6. 获批 Plan 先写失败验收，再做满足结果的最小真源变更；同步真实 writer、reader、测试和派生物，并检查单入口、重复规则、孤儿、平台硬编码与文档预算。
7. handoff 提示词生成不算完成；目标 owner 必须回传**实际变更与 fresh 验证**、残余风险和真实交付状态，来源才可关闭回灌。

## 写入真源

事实、模型推断、提案和用户决策写来源 `findings.md`；实施范围与状态只写目标 owner 的 `task_plan.md`，实际证据写其 `progress.md`。项目事实写项目 owner，用户稳定偏好写用户级现有 owner，skill 改进写目标 skill 的协议 owner。side-task capsule 只是派生输入，不复制实施状态。

## 停止 / 通过

用户已决定提案；暂不做或删除测试未通过时以 no-op 结束。接受的提案必须有唯一 owner、获批 Plan 和可失败验收，目标变更经定向与完整验证 fresh 通过，并已把最终回执交回来源；没有第二真源、孤儿或越权副作用。

## 失败回路 / 能力缺口

推断不足以让用户决策时退回提炼经验补证据；目标事实推翻提案时返回需求澄清或选定方案。writer/reader 无法同步、目标 owner 不可达或外部动作未授权时保持 blocked，不先发布半套协议；发现第二状态 owner 时删除复制，只保留目标 owner。
