# 回灌改进

## 何时进入

复盘产生了有证据的长期改进候选，或正在修改 workflow 本包的入口、reference、template、script、test、生成文档时进入。它负责决定经验进入项目、用户、workflow 还是其他 skill，并保证改进不制造第二真源。

## 已知输入

- 真实失败、重复成本、偏差或用户明确偏好的证据；
- 候选规则的影响面、复现次数、唯一 owner、失败验收和维护成本；
- 当前 package 的 writer、reader、测试、派生物与兼容边界；
- 需要外部写入、发布或跨项目变更的授权状态。

## 深度判断

跨项目晋升必须满足以下之一：

1. **两次独立复现**，证实相同根因与同一通用修复；或
2. **一次严重、系统性且可复现的问题**，并同时具备唯一 owner、失败验收、维护 ROI 为正。

用户明确偏好可进入用户级稳定真源；AI 的推断只留在 findings，直到获得重复证据。项目事实只补项目 owner。会改变其他 workflow/skill 的候选不混入当前产品 Task，必须建立独立进化 Plan 并单独验收。

## 核心动作

1. 写最小问题陈述：触发、实际伤害、已有证据、非范围；单次偶发现象先留候选。
2. 选择唯一 owner。能改现有文件就不新增；没有独立触发与读者问题的内容不得新建 reference 或文档。
3. 先写可失败验收或测试，再做最小机制变更；同步所有真实 writer、reader、测试和派生物。
4. 对本地、可逆、可验证、且完全在用户已确认范围内的真源修正，可自动执行并 fresh 验证。
5. 跨项目推广、其他 skill 修改、发布、删除外部数据或任何外部副作用仍需明确授权；不可因“自动进化”越权。
6. 检查单入口、重复规则、孤儿、深链、平台硬编码、包大小、兼容期限和文档预算。
7. 运行定向门、完整门和 clean-room；收益不再大于维护成本时删除、合并或回退机制。
8. 已获授权且不产生新业务取舍的本地回灌连续完成；只有跨项目推广、外部副作用、责任 owner 冲突或新的难逆决策才交回用户。

## 写入真源

项目事实写项目现有 owner；用户明确偏好写用户级稳定设置；workflow 改进写本包的唯一协议 owner；其他 skill 候选写独立进化 Plan。证据、决策和验证记录进入对应 `findings.md` / `progress.md`，状态只由 `task_plan.md` 持有。

## 停止 / 通过

晋升条件满足，变更有唯一 owner 和可失败验收；定向与完整验证 fresh 通过；没有第二真源、孤儿、越权副作用或负维护 ROI。未达到晋升门时以 no-op 或候选结束，不强行“进化”。

## 失败回路 / 能力缺口

无法证明通用性就退回提炼经验；writer/reader 无法同步则保持 blocked，不先发布半套协议；兼容映射出现第二 owner 时合并到唯一读取边界。需要跨项目或外部动作却未授权时只产出独立进化 Plan，停止写入。
