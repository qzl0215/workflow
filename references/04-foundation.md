# 04 条件｜最小项目地基

## 触发

只有缺口会直接阻碍当前 Request 的安全实施、验证、部署、恢复或后续 AI 交接时触发。已有 owner 足够可用、只是不够漂亮时不加载；不为未知未来建立“大公司式”文档体系。

## 最小上下文与角色

Foundation owner 只接收当前 Request、现有项目入口、真实命令、关键模块/数据流、部署路径、UI 设计 owner 和已证实缺口。先复用并修正现有 README、AGENTS、脚本、配置和设计文档，不另建第二入口。

## 做法

1. 做一次删除测试：如果不补某项也能让当前和下一位 AI 安全找到入口、运行、验证、部署/恢复并理解关键边界，就不建。
2. 最多检查四类地基，按真实缺口选最少项：
   - **AI 项目入口**：一个根入口说明先读什么、项目规则、关键目录和渐进链接；优先补现有 `AGENTS.md` / `README`，不复制全文。
   - **运行与交付入口**：环境前提、bootstrap、run/test/build、必要的 deploy/rollback 命令及 secret 边界；优先让命令可执行，不写长教程。
   - **项目全景导航**：仅在仓库复杂到容易改错边界时，画模块、数据/控制流、外部依赖和 owner 的短地图；链接真实代码，不复制实现。
   - **设计契约**：仅 UI/体验工作需要稳定复用时，补一个 `DESIGN.md` 或现有 design owner，说明 tokens、组件、状态、响应式、动效和无障碍；代码 tokens 仍是实现真源。
3. ADR、术语表、API 目录、数据字典、运维手册等只有当前 Request 有独立读者问题且没有现有 owner 时才建，不作为默认清单。
4. 把每个缺口变成一个有验收的前置 Task，并指定唯一 writer；地基 Task 与产品 Task 按依赖串行，不让多个 Agent 同时改项目入口或共享配置。
5. 为地基 Task 定义 06 阶段的最短 clean-room 验收：下一位 AI 能从入口找到相关模块，成功运行最小验证，并说清部署/恢复或设计边界；05 负责实施，06 按 Task 合同与当前上下文包验收，失败时修唯一 owner，不增加旁路文档。

## 输出与返回

输出“复用/修正/新建/不需要”的最小地基表、唯一 owner、前置 Task、删除理由和 clean-room 验收。计划可安全执行后返回 04 主协议；不要把项目地基扩成独立长期治理项目，除非用户明确把它作为新 Plan。
