# 验收交付

## 何时进入

实现产物已形成，准备声明 Task、Plan 或整体业务结果完成，或准备交付文件、总结、Git 操作、发布、部署、迁移、清理时进入。验收、授权与集成发布是同一动作中的三道连续硬门，不能用授权替代证据，也不能用本地绿灯推定已经发布。

## 已知输入

- 已确认范围、Task/Plan/业务验收和非范围；
- 实际 diff、产物、source fingerprint、运行环境与风险；
- 执行现场、Task 绑定、目标基线、freshness 来源、可写性与脏改动归属；
- 定向、相邻、集成、clean-room 或真实使用验证路径；
- 当前请求、项目规则或 active plan 已经明确授权的目标、范围、remote、branch、环境或外部系统。
- 项目规则、部署/发布文档、CI/脚本、仓库配置、分支保护和真实交付入口。

## 深度判断

按影响面选择 Quick、Standard 或 Full 证据深度：局部可逆任务做定向检查；跨模块或用户路径增加相邻回归和构建；发布、安全、迁移、高风险或多环境任务需要完整门。验证矩阵来自实际变更和业务验收，不照抄历史命令。请求包含交付时，集成发布不是可选收尾，本地已验证只是中间状态。

交付动作按副作用分级：对用户授权范围内的本地可逆文件修改可执行；commit、push、merge、deploy、delete、公开发布和外部写入必须逐目标获得明确授权。当前请求明确要求“提交合并发布”，且项目真源能唯一解析目标时，提交、合并和发布构成一个有边界的授权交付目标，不逐步重复确认；未覆盖目标、目标不明和破坏性例外仍单独确认。

默认用户不审代码：用户只确认业务结果、风险与外部授权，不得要求用户确认代码。AI 负责 diff 自审、范围核对、fresh 验证和机器可读门禁；项目确有独立 reviewer 时由 reviewer 承担代码审阅，但仍不把代码判断转嫁给非技术用户。

## 核心动作

### 证据验收门

1. 从当前验收和实际 diff 建清单；适用时核对红→绿→整理证据。
2. 在声明前 fresh 运行，检查 exit code、完整关键输出、产物 freshness、覆盖范围和环境。
3. 依次通过 Task → Plan → 整体业务；历史绿灯、实现者自报、部分日志和“应该通过”不是证据。
4. AI 总协调者完成 diff 自审并重跑关键路径；存在真实独立 Reviewer 时可增加正交复核，任一失败都回到执行任务、根因回路或上游缺口。
5. 记录未覆盖项和残余风险；不能用低层单测替代真实业务验收。

### 授权交付门

1. 只读确认目标 remote、branch、环境、候选 digest、工作区和敏感信息；不混入无关改动。
2. 向非技术读者先说明业务结果、是否可放心使用、验证和下一步；技术细节只保留复核所需。
3. 只执行已明确授权的副作用；失败立即停止后续写操作，保留可恢复状态和证据。
4. 最终状态按证据报告为本地已验证、已提交、已推送、已合并、已部署或明确 blocked，不得伪造更后一级状态。
5. 面向用户的最终汇报固定补充三行：`代码完成｜`、`合并完成｜`、`线上生效｜`。三项都必须出现；未完成或本次不适用也要直说，不能靠省略让用户猜。
   - `代码完成｜`：只报告已完成、未完成或不适用。
   - `合并完成｜`：报告是否已合并；已合并时说明合入的目标版本名称，例如主版本或项目约定的发布版本。
   - `线上生效｜`：只在真实线上入口通过核验后报告已生效，否则报告未生效或不适用。
   - 用户汇报不得展示提交哈希、commit ID、digest、内部快照名或其他无业务意义的精确字符串；精确技术标识只写入 `progress.md` 等审计真源。
6. 只有需要用户业务验收、补充未覆盖授权或接收最终完成时才交回控制权；可由既有验收和授权自动判定的检查连续完成，不逐层索取确认。

### 集成发布门

1. 按“项目规则 → 部署/发布文档 → CI/脚本 → 仓库配置”的顺序定位交付真源，形成唯一 release contract：remote、目标分支、集成方式、版本/tag/release、部署入口、回滚和发布后 smoke；再生成只保留独立业务价值或技术约束的最小发布图，不凭通用 workflow 猜平台命令。
2. 只纳入当前范围，排除敏感信息、缓存、临时文件和无关改动；候选必须处于可恢复、可审计的分支或等价隔离现场。
3. 获得对应授权后 fetch 最新目标。PR/MR 不是默认步骤，只在分支保护、必需 CI、项目规则或真实 reviewer 明确要求时采用；否则使用可 fast-forward 的直接集成。冲突按双方业务意图合成，禁止 force、整文件选边或绕过分支保护。
4. 目标基线、依赖、冲突解决或候选产物变化时，让受影响旧证据失效；在最终候选和集成结果上完成集成后 fresh 重验，再继续下一外部动作。
5. 按 release contract 执行获准的 commit、push、merge、tag/release、deploy；没有独立价值的节点必须删除，相邻节点能由一个平台动作安全完成时合并执行。任一步失败立即停止后续写操作并保留可恢复现场，不用部分成功冒充完整交付。
6. 发布后核对真实远端与发布状态：目标 ref/commit、tag/Release、CI/部署结果和真实入口 smoke；只有证据存在才报告已提交、已合并、已发布或已部署。
7. 集成或交付完成后记录**现场处置**：仍需完成原范围时为“继续同一 Task”；成果已被目标吸收但现场需留证时为“保留只读”；满足项目清理条件但尚未授权时为“等待授权清理”。已完成现场不得绑定新 Task；不得自动删除、切换、覆盖或丢弃其中内容，清理仍需精确目标、可恢复证据与对应授权。

## 写入真源

release contract、命令、exit code、候选/目标 digest、集成后重验、发布入口、回滚、用户验收、风险、实际交付状态和现场处置写入 `progress.md`；三道门通过后由 `task_plan.md` 更新唯一状态。面向用户的交付摘要基于这些证据生成，不把 changelog 当作业务结果。

## 停止 / 通过

证据验收门覆盖所有纳入范围且 fresh 通过；授权交付门只完成获准动作；请求包含交付时，集成发布门必须完成授权范围内的真实交付并核对状态，或明确 blocked；现场处置与实际交付状态一致。没有外部授权时可以停在“本地已验证”，但不能把任务标为已提交、已发布或已上线。

## 失败回路 / 能力缺口

测试或真实使用失败返回执行任务或根因回路；验收本身错误返回上游动作；缺授权停在 blocker。无 Git、浏览器、部署平台或外部系统能力时仍可交付文件、diff、验证证据和人工下一步，但不得伪造提交、URL 或线上状态。
