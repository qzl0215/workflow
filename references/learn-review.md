# 提炼经验

## 何时进入

交付结果已有证据，或任务被明确终止，需要判断“这次与计划相比发生了什么、下次怎样更少踩坑”时进入。复盘不是必写长总结，而是把真实偏差转成可判断的经验候选。

## 已知输入

- 原目标、推荐方案、Plan/Task DAG、验收与实际结果；
- 失败、回退、重试、人工等待、范围变化和验证记录；
- 实际修改过的项目文档、用户偏好与 workflow 行为；
- 哪些事实有直接证据、哪些只是模型推断。

## 深度判断

按任务分级：

- **轻任务：跳过**；除非发生高风险异常，不建复盘产物。
- **标准任务：条件触发**；出现返工、失败、方向变化、未覆盖风险或明显可复用收益才复盘。
- **项目任务：最小偏差检查**；至少比较计划与实际、发现一个应保留的事实或明确 no-op。

高风险、跨团队、连续失败或候选机制会影响未来多个项目时使用 H2/H3，可安排 fresh-agent 检查；普通项目由当前 owner 完成最小检查即可。

## 核心动作

1. 对照“计划预期 / 实际发生 / 差异证据 / 原因 / 下次触发信号”，不写无证据的成功叙事。
2. 分类经验：项目事实、用户稳定偏好、当前 workflow 缺口、其他 skill 候选、一次性噪音。
3. 问四个高价值问题：什么造成最多返工；哪个早期信号本可更早发现；哪份真源应更新；删掉什么不会损失结果。
4. fresh-agent 检查时只给目标、验收、diff 和最小上下文，让其找遗漏风险与不必要复杂度；不能把 fresh 视角冒充新事实。
5. 对每个候选写事实证据、模型推断的补充需求、影响面、最小改造、唯一 owner、可失败验收、维护成本和预期 ROI。
6. 没有值得沉淀的稳定经验时明确记录 no-op，结束；不为证明流程有价值而制造改进项。
7. 没有需要用户判断的新经验去向或风险接受时，自动完成相称复盘并继续；只有会改变长期偏好、范围或责任 owner 的取舍才交回用户。

## 写入真源

项目独有事实优先短补现有 README、测试、配置或项目文档；候选经验、证据和分类写入 `findings.md`；实际复盘动作写入 `progress.md`。不复制日志、整份对话或模板，不新建“经验大全”。

## 停止 / 通过

回答了“下次更少踩坑”所需的最小问题；每个候选都有证据等级、推断需求和去向，或明确 no-op。复盘篇幅与风险/复用价值相称，没有把模型推断直接写成永久规则，也没有为了完整性无限扩张文档。

## 失败回路 / 能力缺口

缺少实际证据时先补验收或标记未知，不靠记忆推断。候选只对当前项目有效时留在项目 owner；可能形成“小改动、大价值”的补充需求时进入回灌改进，由用户决定是否进入 Plan。无独立 agent 时由当前 owner 做结构化偏差检查，并声明未获得独立视角。
