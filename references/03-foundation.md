# 03 条件｜最小项目地基

## 触发

只有缺口会直接阻碍当前 Request 的安全实施、验证、部署、恢复或下一位 AI 找到正确入口时才触发。已有 owner 足够可用、只是不够漂亮时不补；不为未知未来建立治理体系。

## 最小上下文与责任

只读取当前 Request、现有项目入口、真实命令、关键模块/数据流、交付路径、UI 设计 owner 和已证实缺口。先复用并修正现有 README、AGENTS、脚本、配置或设计文档，不创建第二入口。

## 做法

1. 做删除测试：不补也能让当前和下一位 AI 安全找到入口、运行、验证、交付/恢复并理解关键边界，就不建。
2. 最多检查四类真实缺口：AI 项目入口；可执行的运行/验证/交付入口；复杂仓库的短全景导航；UI 工作需要的稳定设计契约。按当前目标选择最少项。
3. ADR、术语表、API 目录、数据字典或运维手册只有当前 Request 有独立读者问题且没有 owner 时才建，不作为默认清单。
4. 每个缺口成为一个有验收的前置 Task，并指定唯一 writer；地基与产品工作按依赖串行，不让多个 Agent 同时改共享入口或配置。
5. 验收采用最短 clean-room 路径：下一位 AI 能从入口定位相关模块，运行最小检查，并说清交付/恢复或设计边界。失败时修原 owner，不增旁路文档。

## 输出与返回

输出“复用 / 修正 / 新建 / 不需要”的最小表、唯一 owner、前置 Task、删除理由和 clean-room 验收；正式路径写入 `task_plan.md`，简单路径留在当前对话计划。计划可执行后返回 03；除非用户把它明确设为独立业务结果，否则不得扩成长期治理项目。
