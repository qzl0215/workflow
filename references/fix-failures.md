# 修复失败

## 何时进入

出现 bug、测试/构建失败、性能或行为异常、空结果、flaky 状态、真实验收失败，或同一实现路径连续失败时进入。证据已经直接指向单一 typo 或局部语法错误时，可最小修复并做相称验证，不扩成大型诊断。

## 已知输入

- 预期、实际、影响面、最小复现和失败时间；
- 错误、日志、输入、环境、最近变更、调用边界与 source fingerprint；
- 已验证/已排除假设、尝试过的手段和相邻回归；
- 可用观测点、权限、风险和安全回退方式。

## 深度判断

先区分实现缺陷、需求/方案/体验缺口、环境/数据问题和外部 blocker。能够稳定复现且影响局部用 H1；跨层、flaky、数据损坏或多次失败用 H2；生产、安全、不可逆或根因跨系统用 H3。诊断深度由不确定性和风险决定，不由错误表面大小决定。

## 核心动作

1. **写出预期**、实际、影响面和最小复现；不能复现时先增加可观测性。
2. **收集错误**、完整日志、输入、环境、最近变更与调用边界，分开事实和假设。
3. 沿数据/控制流**向上追踪**第一个错误状态，不在末端症状堆补丁。
4. 一次验证一个**可证伪假设**；用最小实验、对照组和条件等待，不增加固定 sleep 掩盖竞态。
5. **修复根因**的最小表面，在正确边界补回归或防御；避免无关重构。
6. 先**重跑复现**，再跑相邻回归和更高层验收；flaky 问题必须证明等待条件和稳定次数。
7. 第一次失败定位证据；第二次换本质不同观察点或实现；第三次列假设、反转假设并扩大源码/文档证据，必要时请求决策。

## 写入真源

复现、假设、实验、根因、修复、命令、exit code 和残余风险写入 `progress.md`；新的稳定项目事实补回原代码、测试、配置或 README owner。临时日志和实验脚本在证据记录后移除，不建独立故障知识库。

## 停止 / 通过

根因能够解释现象；最小修复使复现转绿；相邻回归与上层验收通过；未覆盖风险透明。只能消除症状却不能解释根因时保持未完成，除非用户明确接受有证据的临时缓解与退出条件。

## 失败回路 / 能力缺口

事实表明不是 bug 时返回产生缺口的上游动作。同一阻断重复出现时保留排除证据并换工具、观察点或路线。危险诊断、生产写入或新权限必须停止并请求授权；缺少日志/环境时写具体 blocker，不让用户笼统“手动看看”。
