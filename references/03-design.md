# 03｜选定设计

先画清 90 分最终画面，再收敛最小充分方案；本阶段解决方向、体验和系统边界，不写执行排期。

## 最小上下文

- 已确认的用户结果、Requirement Ledger、关系图和业务验收；
- 带来源的事实、约束、不变量、非范围、可验证假设和 blocker；
- 现有产品、数据、技术、体验、维护和交付边界；
- 用户对质量、速度、风险和可逆性的稳定偏好。

不加载执行日志、Task 状态、Git 交付细节或所有淘汰方案历史。

## 责任角色

**方案 owner**对端到端推荐和删除项负责；动态专家只提供独立判断，不写正式真源。涉及 UI 时由**体验 owner**先完成旅程、结构、视觉和动效选择，再交给 Plan owner。

## 阶段能力

1. 从最终用户结果倒推能力、数据、身份、平台、生命周期、约束和风险，形成完整森林而非局部功能清单。
2. H2/H3、多路线、跨域、难逆、局部最优可能伤害全局，或用户明确要求专家发散时加载 `references/03-expert-forest.md`；H0/H1 不为形式召集专家。
3. 存在页面、旅程、视觉、交互、动效、响应式或无障碍方向取舍时加载 `references/03-experience.md`，并在计划前收敛；不因文案 typo 或既定样式的小修自动升级。
4. 形成 2–3 条端到端可行路线，比较用户价值、复杂度、维护成本、可逆性、风险和验证速度，只保留一个推荐及确有价值的备选。
5. 做删除测试：移除某项不损害用户结果或关键风险，就不进入推荐方案。
6. 把剩余未知转为事实、明确选择、可失败实验或 blocker，不能留给 Build 猜。

## 输出

- 90 分最终画面、推荐方案和必要备选；
- 关键取舍、删除项、系统/数据/体验边界和不变量；
- UI 体验契约（适用时）、迁移/回滚点和风险处置；
- `findings.md` 中的方案比较与决定依据，以及供用户确认的 `plan.md` 完整方案草案。

## 退出与路由

推荐能解释为什么值得做、为什么不是更小方案，方向性未知已闭环后路由到 **04 计划编排**。事实不足返回 02；用户取舍未定返回 02 的 Grill；UI 方向未定继续本阶段体验协议。不得用“实施时再定”通过。
