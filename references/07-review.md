# 07｜复盘升级

比较预期与实际，只留下能减少下一次返工的最小事实或机制；不为证明流程有价值而制造长总结、额外文档或改进清单。

## 最小上下文

- 原目标、已选方案、Plan/Task 图、验收和最终交付结果；
- 真实失败、返工、等待、范围变化、反馈、冲突与残余风险；
- 实际修改过的项目真源和用户稳定偏好；
- 长期改进候选的复现次数、影响面、失败证据和维护成本。

不加载原始聊天或全部日志；只引用能解释偏差和复用价值的证据。

## 阶段能力

默认把本阶段压成最终汇报前的三个快速判断：是否仍未完成、是否有已证实的高价值复用、是否有步骤可以删除。三者均否时直接 no-op，不派 Agent、不新建文档、不延迟交付。

1. 对照“预期 / 实际 / 差异证据 / 根因 / 下次触发信号”，寻找最多返工、最早可见信号、应更新的唯一 owner 和可以删除的步骤。
2. 区分项目事实、用户稳定偏好、workflow 缺口、其他能力候选和一次性噪音；不同类型进入各自 owner，不混成第二真源。
3. 本地、可逆、当前授权范围内的项目事实修正可以由对应 owner 更新并 fresh 验证；没有稳定收益时明确 no-op。
4. 候选准备进入长期机制、跨项目推广或修改本包时，使用 `references/07-evolve.md`；一次顺利或偶发现象不能直接晋升。
5. 复盘发现交付仍未满足验收时，返回产生缺口的阶段继续完成，不能用复盘文字掩盖未完成。
6. 已有 `progress.md` 最终只保留当前业务结果、验收结论、残余风险和明确下一步；简单路径不为复盘新建文档。已处理反馈、等待与过期记录移除，历史交给 Git 或原始产物。

## 输出与结束

- 预期/实际差异、返工根因、删除项和下次触发信号；
- 项目或用户真源的最小修正与 fresh 结果，或 no-op；
- 长期改进候选的证据、owner、失败验收和维护收益；
- 面向用户的业务结果、验证、真实交付状态、残余风险和下一步。

每个候选有明确去向，项目修正已验明，或确认没有值得沉淀的稳定经验后结束。需求、方案、计划、实现、验收或交付仍有缺口时分别返回 01–06；长期改进达到门槛时进入本阶段受控进化。
