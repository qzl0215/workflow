# 06 fallback｜Git 集成与冲突修复

只有 Request 候选需要 commit、同步目标、rebase、解决冲突、push、创建 PR/MR 或 merge，且没有用户指定或宿主原生 Git 集成能力时才读取。工作现场已由 ◆开工门确认；若当前动作要求更强隔离，先返回开工门升级，不在本协议私自另建现场。

## 最小输入

Request 工作现场与实际隔离方式、目标 remote/ref、候选 diff、逐文件意图、双方不变量、验证矩阵、项目规则和逐项授权。不得仅凭提交标题猜业务意图。

## 最小方法

1. 只读检查候选 diff、未跟踪/敏感文件、项目提交规则和当前授权；只纳入本 Request。
2. 获得相应授权后 fetch 最新目标跟踪引用；目标前进时将 Request 分支 rebase 到该引用，不从旧本地分支推断最新状态。
3. 文本冲突读取共同祖先、目标版本、Request 版本、相邻代码、双方意图和测试；合成同时保护不变量的实现，不整文件选择一边。
4. 文本无冲突但验证失败按语义冲突处理。修复后继续 rebase，并 fresh 运行受影响 Plan、跨 Plan 集成和 Request 关键路径。
5. 目标再次前进则重新同步。不得 force push 目标分支；私有 Request 分支也只有项目允许且用户已授权时才考虑安全租约式更新。
6. 双方业务意图真实互斥且无法从事实判断时，列出冲突、已排除方案、推荐与备选并请求决定；不得以 abort/reset 或逃避分支伪装解决。

## 输出与返回

输出最终 commit/digest、最新 base、冲突与语义合成、集成后命令和 fresh 结果，以及实际 push/PR/MR/merge 状态。证据齐全后返回 06 主协议；失败时保持可恢复现场，并把实现、验收或业务决定分别路由到 04、05 或 01–03。
