# 06｜集成交付

把所有已验收 Plan 汇合成 Request 候选，基于最新目标证明整体结果，并准确报告本地、Git 或线上真实状态；本阶段守住授权、整体完成和失败路由，不与宿主争夺 Git 或发布方法。

## 最小上下文

- 全部可汇合 Plan、候选 digest、Request 工作现场、目标环境或 remote/ref；
- 逐文件/模块变更意图、关键不变量、整体验证矩阵和回滚方式；
- 05 的验收记录、source fingerprint、敏感信息检查和残余风险；
- 用户对当前具体目标已明确授予的副作用权限。

不加载完整 Grill、方案发言、工作对话或无关 Plan 证据。

## 方法 owner

Git、发布和部署分别选择一个用户指定或兼容的宿主原生能力。命中后不执行、不叠加同类条件 fallback，但仍必须遵守本文件授权、最新基线、fresh 验收和真实状态规则。需要 Git 集成而宿主无对应方法时，才读取 `references/06-git-integrate.md`。

## 阶段能力

1. 只读确认所有纳入 Plan 已通过、候选 diff 不含无关改动、目标与项目交付规则明确，并为 commit、push、merge、deploy、delete、公开发布、生产写入等动作逐目标检查授权。
2. 单个 Plan 通过不能提前集成共享分支；只有全部纳入 Plan 可汇合后才执行 Request 级同步、冲突处理和必要的整体路径验证。
3. 未变化的 Task/Plan 证据不因进入 Request 层级重跑。目标基线前进、rebase、冲突修复、依赖或产物变化会改变 fingerprint；只失效受影响验收，并 fresh 验证跨 Plan 集成、Request 关键路径及实际受影响的 Plan。
4. 文本或语义冲突必须依据共同祖先、双方业务意图、约束、相邻代码和测试合成，不能整文件选边、force、reset 或建逃避分支掩盖问题。
5. 未授权动作停在可恢复现场；交付失败立即停止后续写操作，记录已完成层级、失败证据、回滚方式和安全下一步。
6. 面向用户先说明业务结果、能否放心使用、验证、真实交付层级和下一步；技术细节只保留复核所需。
7. 结束前只做三个快判断：是否仍有未完成结果，是否出现反复返工或可删除步骤，是否有达到复用门槛的经验。全部为否时在最终汇报记为 no-op 并直接结束；任一为是才读取 `references/07-review.md` 进入 07。

## 输出与路由

- 最终 diff/产物 digest、目标基线和集成后 fresh 结果；
- 冲突双方意图、解决方式与验证；
- 本地已验证、已提交、已推送、已合并、已部署或等待授权的准确状态；
- 回滚方式、残余风险，以及已有 `progress.md` 的最终当前切片；简单路径留在对话。

整体结果基于最新目标通过、授权内动作完成且状态准确后，按上方三个快判断直接结束或进入 **07 复盘升级**。验证失败回 05 或将实现缺陷回 04；需求、方案或计划错误分别回 01、02、03；意图互斥且事实不足时回对应用户决定点；缺授权时保持可恢复现场。
