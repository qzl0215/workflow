# 并行成果合并

## 何时进入

同一仓库有两个以上独立线程准备交付，目标分支在执行期间前进，或 rebase 出现文本/语义冲突时进入。它只治理跨线程成果如何安全落到同一目标分支；当前 Plan 内如何分派 Agent 仍由协调 harness 负责。

## 已知输入

- 每个线程的独立 worktree、分支、提交、目标分支与 remote；
- 项目级提交/发布规则、当前请求或 active plan 中已有的长期授权；
- 每个线程的 MERGE_NOTE：目标、文件意图、关键不变量和验证命令；
- 最新目标分支、共同祖先、文本冲突文件与可能受影响的业务路径。

## 深度判断

共享 workspace 内同时编辑同一文件必须停下重新分域；不同线程位于独立 worktree 时，允许修改同一原始文件，开发期不做悲观文件锁。并行收益来自开发隔离，安全性来自合并队列串行落地、后合并者兼容先合入成果，以及文本与语义两层验证。

授权先读项目规则，再应用通用门：项目发布契约决定是否使用 CI 或 MR，通用 workflow 不擅自把一种交付模式推广到所有项目。项目规则、active plan 或当前请求已经给出的长期授权，在更高优先级运行时政策允许时属于明确授权，不重复向用户索取同一授权；未覆盖的 remote、branch、环境或破坏性动作仍停在授权门。

## 核心动作

1. 每条修改线程创建前先 fetch 最新目标分支，再从该 remote tracking ref 建立独立 worktree 与分支，尽早小步 commit；不得从本地可能过期的目标分支或另一条功能分支开始。项目发布契约决定是否使用 CI 或 MR；同一原始文件可并行开发，但同一时刻只允许一条线程进入合并队列。
2. 合并前补 MERGE_NOTE：改动目标与动机、逐文件意图、关键不变量、本线程验证命令；缺失时先从实际 diff 和计划补齐，不凭提交标题猜。
3. 后合并者 fetch 最新目标分支并 rebase；无冲突路径可运行 `python3 scripts/safe_merge.py --target <branch> --verify "<command>" --push`，它负责本机共享锁、rebase、验证、fast-forward 校验和被超车重试，绝不 force。
4. 文本冲突时机械件以退出码 3 停下并保留冲突现场。AI 读取双方 MERGE_NOTE、冲突块、相邻代码和历史约束，比较双方的前提假设、时间先后、不变量及受影响测试，保全兼容意图后 `git add <files>`，再运行同一命令并增加 `--continue`（等价于受控执行 `git rebase --continue`）；不得整文件选边；不得 rebase --abort、reset 到目标分支或新建 `-r2` 分支来逃避正常消解。
5. rebase 完成后运行本线程验证命令，并运行受影响的已合入线程的双方验证命令；文本无冲突但验证失败视为语义冲突，修复后重新验证。
6. 目标分支再次前进时重新 fetch/rebase；推送前必须证明目标分支是候选提交祖先。项目要求安全提交工具、MR 或发布命令时先走项目规则，机械件不得绕过。
7. 只有双方意图真实互斥、无法从事实判断时才升级：列双方意图、冲突点、已排除方案、推荐与备选；释放合并队列，让其他线程继续。

## 写入真源

MERGE_NOTE 写入当前线程既有 `findings.md`，实际 rebase、冲突、取舍、命令、exit code 和验证写入 `progress.md`；范围与状态仍只由 `task_plan.md` 持有。合并锁位于 Git common dir，只是瞬时协调状态，不进入版本化真源。

## 停止 / 通过

当前线程已基于最新目标分支；文本冲突按双方意图消解；本线程和受影响线程的验证均 fresh 通过；目标分支只接受 fast-forward 更新；实际提交、推送、合并、发布层级与授权和证据一致。主线保持可工作，后续线程可从它继续。

## 失败回路 / 能力缺口

退出码 3 保留现场给 AI 继续；验证失败回到语义冲突定位；连续被超车达到上限时释放锁并稍后重排，不 force。缺 MERGE_NOTE 时从 diff、计划和测试重建最小意图；无可执行验证时只能完成文本合并与影响分析，不把它描述为语义安全。冲突无法兼得时升级用户，但不阻塞其他合并线程。
