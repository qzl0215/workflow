# 拆成任务

## 何时进入

推荐方案、边界和验收已经选定；UI 范围已有体验结论；需要把业务结果拆成可执行 Plan/Task、串并联关系、owner 和验证路径时进入。没有确认方向时不得用计划掩盖设计缺口。

## 已知输入

- 目标、推荐方案、非范围、约束、不变量和风险；
- 业务验收、体验结论、回滚点与授权边界；
- 真实目录、文件 owner、运行/测试命令和共享资源；
- 已有 `task_plan.md`、`findings.md`、`progress.md` 及必要的技术导航。

## 深度判断

用**P9 六要素**判断每个工作单元是否可执行：**目标、输入、输出、验收、文件域、依赖**。缺任一项就不是 Ready Task。Plan 按可独立验收的业务结果分组；Task 按单一 owner、可隔离文件域和 fresh 验证分解。

估时不套用人的工作天数。分别评估 AI 执行路径、人工决策/验收等待、外部依赖、共享资源排队与关键路径；绝对时间缺少实测依据时给相对复杂度和 blocker，不制造精确幻觉。

## 核心动作

1. 先写一页决策摘要、现状、推荐范围、非范围和可观察验收。
2. 给每个 Plan 标注`来源：原始需求 / 回灌提案 / 外部 handoff`。原验收缺口留在原 Plan；**新的用户结果追加新 Plan**，不能塞进已完成 Task。
3. 外部 handoff 已确认需求与方案时只验证目标项目最易过期的事实，不重复追问；新事实推翻输入才返回需求澄清或选定方案。
4. 建 **Plan 总 DAG** 表达业务结果之间的先后关系；每个 Plan 内建 **Task 局部 DAG**，明确分叉、汇合和 Ready Queue。
5. 为每个 Task 填 P9 六要素、owner、共享资源、授权门、风险、回退和实际验证命令。
6. 只有两个以上 Task 依赖已满足、文件域隔离、验证独立且并行收益大于协调成本时并行；共享文件、发布候选或顺序依赖必须串行。
7. Task ID 保持稳定；实施状态只在目标 owner 的 `task_plan.md` 持有。复杂实现才写 `implementation-plan.md`，且只做技术导航。
8. 路径、命令或验收未知时写 blocker；不得残留 TODO、虚构路径或把会改变方案的问题下放给 Task owner。
9. 用户按可独立验收的整个 Plan 确认纳入/暂缓/不做；Task 用于透明编排，不逐项索取无意义批准。
10. 只有**完整 Plan 确认**需要交回用户；Task 拆合、依赖排序、文件域和验证路径等不改变业务契约的编排由 AI 自行收敛，不制造逐项确认。

## 写入真源

`task_plan.md` 保存来源、范围、Plan/Task DAG、P9 六要素、owner、验收与唯一状态；`findings.md` 保存事实和取舍；`progress.md` 保存实际动作与证据；复杂任务的 `implementation-plan.md` 只保存实现导航。跨 owner 的来源任务只保留提案决策、目标入口和最终回执，不复制实施状态。

## 停止 / 通过

整个 Plan 已获确认；每个 Ready Task 都能在不重新设计的情况下回答做什么、从哪里开始、产出什么、谁负责、改哪些文件、依赖谁、如何 fresh 验收。依赖 DAG 无隐藏环，文件域冲突已串行化，外部副作用仍有单独授权门。

## 失败回路 / 能力缺口

目标或验收变动返回需求澄清；方案或体验仍需选择返回选定方案。仓库事实不足时只读补证据，不猜文件。无法安全隔离的 Task 合并为同一 owner 串行执行；无并行能力时按 Ready Queue solo 推进，验收不变。
