# 需求澄清

## 何时进入

标准或项目任务开始时进入；目标、用户结果、事实、优先级、验收、非范围、授权或项目入口任一会影响方向却仍不清楚时继续停留。用户最初说出的需求只是待验证输入，不因 AI 已能写出一张看似完整的需求卡而视为澄清完成。轻任务已有可信输入和可观察结果时，可记录依据后快速通过，不为追问而追问。

## 已知输入

- 用户原始请求、已表达偏好与业务背景；
- 项目规则、README、配置、入口代码、测试、历史计划和可用工具；
- 已证实事实、带来源的推断、约束、现有真源与能力边界；
- 谁为结果负责、什么外部动作需要授权；
- 已回答问题、仍开放的决策分支、用户明确延期项与当前理解快照。

## 深度判断

先做**价值门 A**：这件事解决谁的什么问题，结果为何值得投入 AI、什么信号说明值得继续。再问一句：如果现在进入下一动作，是否必须猜关键事实，或无法验证结果？

把未知只分为四路：

- **事实可查**：可从项目、数据、工具、配置或可信来源获得；AI 自己查。
- **取舍待定**：价值、优先级、风险接受度或主观方向需要决策 owner 选择。
- **假设待验**：继续讨论不会变成事实，应以实验、原型、小样或可失败验收验证。
- **外部待解**：依赖权限、凭据、外部 owner、生产事实或明确授权。

再过**需求成熟度硬门**，逐项检查五个承重分支：

1. **用户 / 场景 / 现状痛点**：谁在什么情况下遇到什么问题，当前如何替代，为什么现在值得改变；
2. **目标结果与可观察成功标准**：做成后谁能完成什么，什么证据说明有效；
3. **范围与非范围**：本次必须覆盖、明确不做和可以延期什么；
4. **关键约束、优先级与取舍**：时间、质量、成本、体验、风险冲突时先保什么；
5. **授权边界与责任人**：谁能决定，哪些外部动作、数据或风险需要授权。

每个分支必须有可追溯处置：事实有来源，主观取舍有用户回答，真假未知有可验证假设，范围外依赖有明确 blocker。会改变目标、范围、验收、优先级或授权的判断必须由用户作答或明确延期；AI 可以给推荐，不能把推荐伪装成用户决定，**沉默不等于确认**。只有局部、可逆且不改变结果的实现细节才可采用已声明默认。

**需求卡是澄清完成后的结晶，不是需求挖掘工具。** 需求成熟度硬门未通过时，禁止生成完成态需求卡、登记需求澄清成果或进入下一动作；可以展示当前理解和开放问题，但必须显式标为待确认。复杂度只决定调查深度，不降低两道门。能只读发现的事实不问用户；只有不能自行发现的关键问题才提问。

## 核心动作

1. 按最短路径检查项目根、相关模块、规则、真实入口、必需命令和验证路径；把事实、推断和未知分开并标来源，不做全仓资料考古，也不把可查事实重新抛给用户。
2. 把请求改写为用户结果，建立承重问题的**决策树**：先列顶层分支，再标出独立、依赖、互斥、共享基础和当前处置；优先追问最可能改变目标、范围、验收、优先级或授权的开放分支。
3. 为每个关键问题保留稳定编号、它会改变的结果、推荐答案、理由、主要代价和 2–3 个互斥选项。**回答含糊**、回避取舍或与前文冲突时，指出缺口并沿用**同一编号**继续追问；不得自行补成确定需求。
4. 使用**编号问题批次**提高效率：相互独立、同层且用户可一起判断的问题通常每批 2–5 个；**有依赖**的问题只问当前最上游分支，得到答案后再生成下一批。用户可按 `1B 2A 3C` 回复，也可在序号后补充说明。批量不等于问卷倾倒：每批只放当前最高价值问题，不一次展开整棵树。
5. 收到回答后更新决策树：移除已解决问题，追加答案暴露的新分支；“更快、以后、做好一点、都可以”等非具体答案必须转成指标、边界、优先级或明确默认后才算回答。不设置总问题数上限，继续追问直到所有承重分支已解决、可验证、明确延期或成为 blocker。
6. 使用**短反馈回路**：每批前简述当前理解与本批影响，每批后总结已确认、仍开放和正在深挖的分支；每形成一个会改变目标、范围、验收或方向的关键理解，就让用户尽快纠正。只有用户已在原请求或可信真源中明确回答全部承重分支时，才可零追问快速通过。
7. 两道门通过后才整理完成态需求卡或等价成果；信息存在但分散或过期时优先补现有 README、规则或真源。没有 owner 且缺口确实阻断时，最多一份短入口文档；只做导航，不复制部署、安全、设计或运行细节。

## 写入真源

事实、来源、推断、四路未知、价值门 A、需求成熟度硬门、决策树、问题编号、用户回答、目标、验收、非范围和授权边界写入现有 `findings.md`；项目任务只在两道门通过后同步 `task_plan.md` 的已确认范围。成熟项目默认 no-op；已有 owner 就原位补齐，不创建同义说明。文档预算是 0，只有无 owner 的阻断入口例外为 1。

## 停止 / 通过

价值门 A 与需求成熟度硬门均已通过；五个承重分支都有来源明确的事实、用户回答、可验证假设或 blocker，不再存在被 AI 默认替代的关键用户判断。用户结果、优先级、可观察验收、非范围和授权边界足够，下一动作无需猜关键事实；继续调查或追问已不再改变目标、范围、顺序、主要风险、验收或授权时停止。无业务价值、证据证明无需做或已有结果满足时，允许以 no-op 结束。

## 失败回路 / 能力缺口

事实不可得时转为假设待验或外部待解，不编造。存在关键取舍、含糊回答、矛盾或用户要求压力测试时必须加载决策挑战 harness；用户暂不回答时，只能对安全可逆的局部项采用已声明默认，关键判断则缩小范围、明确延期或保持 blocker。缺凭据、权限或高风险授权时写清 blocker、外部 owner 和解锁条件。无网络、浏览器或 memory 时改用仓库与当前对话证据并标未覆盖项，不把能力缺口伪装成事实。
